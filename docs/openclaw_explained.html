Section 01

The Big Picture

OpenClaw is one program — called the Gateway — that runs on your computer 24/7. It connects to all your messaging apps, listens for messages, sends them to an AI, and replies on your behalf.

🏠
Think of it like a smart home hub. Just like a Philips Hue hub connects your phone to your lightbulbs, OpenClaw's Gateway connects your messaging apps (WhatsApp, Telegram, Discord…) to AI models (OpenAI, Claude, Gemini…). The hub is always on, always listening, and routes signals between devices.

Section 02

Follow a Message — Step by Step

Let's trace exactly what happens when you send "What's the weather in Paris?" via WhatsApp and get an AI reply back. Click Next Step to walk through it.

📱

Step 1 — You send a WhatsApp message

You type "What's the weather in Paris?" and hit send on your phone. WhatsApp delivers it to OpenClaw's WhatsApp connection (powered by the Baileys library running inside the gateway). This works because OpenClaw scanned a WhatsApp Web QR code when you set it up.

📡

Step 2 — Channel Manager receives the message

The Channel Manager gets the raw message. First it checks: Is this sender allowed? If you're already paired (you've previously approved this contact), the message passes through. If not, the sender gets a pairing code instead of an AI reply.

incoming message from +33612345678 → check allowlist ... PASS → dmPolicy = "pairing" → sender approved ✓ → route to agent session
🔌

Step 3 — Plugin System activates the right tools

The Plugin System looks at your config and figures out which plugins are needed for this request. For a weather query it might activate a web search plugin (like Brave Search or Tavily). Each plugin declares what it can do via a manifest file.

trigger: { kind: "capability", capability: "web-search" } → activating plugin: brave-search → activating plugin: anthropic (claude-3-5-sonnet) → ready
🤖

Step 4 — Agent Runtime processes the request

The Agent Runtime (built on Pi agent core) takes over. It loads your workspace files (AGENTS.md, SOUL.md etc) to understand how you want the AI to behave. Then it sends the message + conversation history + available tools to the AI model.

session: abc123.jsonl (loading 14 prior messages) injecting workspace: AGENTS.md, SOUL.md, TOOLS.md calling model: claude-3-5-sonnet-20241022 tools available: [brave_search, read, write, bash...]
✨

Step 5 — AI model reasons and uses tools

The AI model (Claude, GPT-4, Gemini, etc.) sees your message and decides to use the brave_search tool to look up current weather. The result comes back, the model reads it, and formulates a natural-language reply. All tool calls stream back to the agent in real time.

model → tool_use: brave_search("Paris weather today") tool → result: "Paris: 18°C, partly cloudy, wind 12km/h" model → text: "The weather in Paris right now is 18°C and partly cloudy with light winds of 12 km/h. Perfect weather for a walk! ☀️"
📨

Step 6 — Reply delivered back via WhatsApp

The Agent Runtime sends the final reply back through the Channel Manager. The WhatsApp channel plugin formats it for WhatsApp (markdown → WhatsApp format) and sends it. The full exchange is saved to your session file. Total time: ~2–4 seconds.

saving to: sessions/abc123.jsonl → formatting for whatsapp channel → sending: "The weather in Paris right now is 18°C..." → delivered ✓ (2.3s total)

The Gateway
The core daemon process

The single program that runs everything.

The Gateway is the always-on server process that runs on your machine. You start it with openclaw gateway and it stays alive in the background (installed as a system service via launchd on macOS or systemd on Linux).

It listens on port 18789 and handles:

  • Maintaining connections to all messaging platforms
  • Running the AI agent for each conversation
  • Serving the WebChat UI and Canvas
  • Authenticating connected devices (macOS app, iOS, Android)
  • Running scheduled (cron) tasks

One Gateway per machine. All apps connect to it over WebSocket.

TypeScript Node.js port 18789 WebSocket
+
📡
Channels
Messaging platform connectors

How OpenClaw talks to WhatsApp, Telegram, Discord, etc.

Each messaging platform is a channel extension — a plugin that knows how to connect to that specific platform's API. There are 25+ supported channels.

  • WhatsApp → Baileys library (WhatsApp Web protocol)
  • Telegram → grammY library (Bot API)
  • Discord → Official Discord bot API
  • Signal → signal-cli / native integration
  • iMessage → BlueBubbles or native bridge
  • Slack, Teams, Matrix, IRC, LINE, WeChat…

Each channel plugin registers itself with the Gateway at startup. The Gateway then routes inbound messages to the agent and outbound replies back through the right channel.

25+ channels extensions/
+
🤖
Agent Runtime
The AI thinking engine

Where the actual AI reasoning happens.

Built on Pi agent core (@mariozachner/pi-agent-core) — a third-party agent harness library. OpenClaw wraps this with session management, tool wiring, and channel delivery.

On the first turn of a new conversation, it loads your workspace files:

  • AGENTS.md — how you want the AI to behave
  • SOUL.md — the AI's personality and tone
  • TOOLS.md — your notes on tool usage
  • IDENTITY.md — the AI's name and emoji
  • USER.md — facts about you

These files are just plain text you can edit. They act as the AI's "personality" and "memory" for every new conversation.

pi-agent-core src/agents/

🔌
Plugin System
Manifest-driven extension loader

How OpenClaw loads only what it needs.

Each plugin has a manifest — a JSON file declaring what it provides: a channel, a model provider, a tool, or a command. The PluginActivationPlanner decides which plugins to load based on a trigger (what the current request needs).

Two plugin styles:

  • Code plugins — full TypeScript runtime hooks, providers, tools
  • Bundle-style plugins — packages of skills, MCP servers, config files

Community plugins are published to ClawHub — the plugin registry. You install them like npm packages.

100+ plugins extensions/ src/plugins/

💾
Sessions
Conversation memory

How the AI remembers what you said.

Every conversation is a session. Sessions are stored as JSONL files (one JSON object per line) at:

~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl

Each line in the file is one message or AI reply. This is how the AI "remembers" the conversation — it reads the file at the start of each turn.

  • Different channels can map to isolated agent sessions
  • Sessions have configurable pruning (auto-delete old ones)
  • You can inspect, export, or share session files manually
  • The /new chat command starts a fresh session
JSONL format src/sessions/

🧠
Memory System
Long-term knowledge store

How the AI remembers things across conversations.

Sessions store a single conversation. Memory stores facts across all conversations. When you tell the AI "my daughter's name is Sophie", memory saves that and recalls it next week.

Three memory backends (pick one):

  • active-memory — simple append-to-file, fast, no setup
  • memory-lancedb — vector embeddings, semantic search ("find stuff I said about cooking")
  • memory-wiki — wiki-style structured notes

Only one memory plugin can be active at a time (configured in your gateway config).

extensions/active-memory LanceDB vectors

🔧
MCP (Tool Bridge)
Model Context Protocol

The standard protocol for AI tools.

MCP (Model Context Protocol) is an open standard by Anthropic for connecting AI models to external tools. OpenClaw plays both roles:

  • MCP Server — exposes OpenClaw's tools (bash, read, write, browser…) to other AI apps via stdio
  • MCP Client — connects to external MCP servers (e.g. a database tool server, a GitHub tool server)

This means OpenClaw is compatible with the growing ecosystem of MCP tools. Any MCP tool server can be plugged into OpenClaw without writing custom code.

open standard src/mcp/

Canvas
AI-driven live visual workspace

The AI can draw UIs in real time.

Canvas is an HTML page served by the Gateway at /__openclaw__/canvas/. The AI can write HTML, CSS, and JavaScript to this page and you see it update live.

Use cases:

  • Interactive dashboards the AI builds on demand
  • Charts and visualizations from your data
  • Custom forms and calculators
  • Live document editors

A2UI (Agent-to-UI) is the protocol for the AI to interact with Canvas elements — it can read button clicks and form values back from the page.

live HTML/CSS/JS A2UI protocol

Cron Scheduler
Automated recurring tasks

Schedule AI tasks like a calendar.

The Gateway has a built-in cron scheduler using the croner library. You can tell the AI to run tasks on a schedule — and it will keep running them even when you're not around.

  • "Every Monday at 8am, send me a news digest on WhatsApp"
  • "Daily at 6pm, check my calendar and send a reminder"
  • "Every hour, check if my website is down"

Cron jobs are stored in your gateway config and survive restarts. You can view active cron jobs with openclaw cron list.

croner library src/cron/

Voice System
Wake words, speech, and TTS

Talk to your AI out loud.

Voice is handled by a set of extensions that work together:

  • Wake word detection — always-on listening on macOS/iOS
  • Speech-to-text — Deepgram (cloud) or local Whisper
  • Text-to-speech — ElevenLabs (natural), Azure TTS, or system TTS
  • Android — continuous voice mode, always listening
  • macOS MLX TTS — fully on-device TTS using Apple Silicon

On macOS, the Push-to-Talk overlay activates when you hold a hotkey. On Android, you can speak freely and it listens continuously.

Skills
Pre-packaged AI workflows

Like apps for your AI assistant.

Skills are pre-written instruction files that teach the AI how to do specific things. They're loaded from your workspace skills/ folder or from bundled defaults.

  • coding-agent — full software engineering workflow

WebSocket Protocol
How everything communicates

The wire format for the control plane.

All clients (macOS app, iOS, Android, CLI, WebChat) connect to the Gateway over a WebSocket on port 18789. The wire protocol uses JSON frames.

// Request frame { "type": "req", "id": "abc", "method": "agent", "params": { "message": "What's the weather?" } } // Response frame { "type": "res", "id": "abc", "ok": true, "payload": { "runId": "xyz", "status": "accepted" } } // Streaming event frame { "type": "event", "event": "agent", "payload": { "text": "The weather in Paris is..." } }

The first frame must always be connect with a signed challenge to prove device identity. After that, you send requests and receive responses and events.

The Plugin Ecosystem

Section 05

Security Model

OpenClaw is powerful — the AI can run bash commands, read/write files, browse the web. So security defaults matter a lot. Here's how it's designed to be safe.

🔐

Device Pairing (Zero Trust)

Every device that connects to the Gateway must prove its identity with a signed challenge. New devices get a pairing code. The Gateway keeps a paired-device store. Even local connections on the same machine must pair (unless loopback auto-approve is on).

🚪

DM Pairing Policy

By default, unknown senders via WhatsApp, Telegram, Discord etc. receive a short pairing code instead of AI access. You approve them with openclaw pairing approve. This prevents strangers from talking to your AI. You can set dmPolicy="open" to allow anyone.

📦

Sandboxing Non-Main Sessions

Your personal "main" session runs with full tool access. Sessions from group chats or untrusted channels can be sandboxed in Docker containers — so the AI can still run code but can't touch your personal files. Set agents.defaults.sandbox.mode: "non-main".

🏠

Local-First by Default

The Gateway binds to 127.0.0.1 (localhost only) by default. It is not reachable from the internet unless you explicitly set it up (via Tailscale, ngrok, SSH tunnel, etc.). This means your AI is private by default — no cloud server holds your conversations.

⚠️

Main Session Has Full Access

When it's just you talking to your AI, the agent runs with full host access — it can read files, run commands, browse the web. This is intentional for a personal assistant. If you expose the gateway to others, use sandboxing or restrict tool policies.

🔑

Run openclaw doctor

Before exposing your gateway remotely, run openclaw doctor. It scans your config and warns about risky settings — like dmPolicy="open" on a public-facing gateway, or gateway auth mode set to "none" without a firewall.
