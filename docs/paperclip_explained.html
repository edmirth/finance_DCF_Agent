The Big Picture

What is Paperclip?

Before diving into code, let's understand the mental model.

Imagine you want to build a software startup — but instead of hiring humans, you hire AI agents. You'd still need a CEO, a CTO, engineers, and marketers. They'd still need tasks, deadlines, budgets, and someone to approve big decisions.

Paperclip is the operating system for that company. It handles org charts, task assignment, budgets, approvals, and 24/7 autonomous operation — all for AI agents instead of humans.

— The core idea behind Paperclip

💡 Think of it this way: OpenClaw / Claude Code is a single employee. Paperclip is the entire company that employs them.

Core Concepts

The 8 Building Blocks

Everything in Paperclip is built from these 8 ideas. Click each tab to explore.

🤖 What is an Agent?

An agent is an AI worker in your company. Each one has:

  • Name & Title — e.g. "Alice, Senior Engineer"
  • Role — A job description written in plain text
  • Adapter type — Which AI tool it uses (Claude, Codex…)
  • Monthly budget — Max it can spend in tokens/API costs
  • Reports to — Its manager (another agent or you)
  • Instructions — A markdown file it reads before each run

📋 Agent in the Database

Each agent is one row in the agents table:

id: "agt_abc123"
name: "Alice"
title: "Senior Engineer"
role: "You build features..."
adapterType: "claude_local"
budgetMonthlyCents: 5000
reportsTo: "agt_cto456"
status: "active"

📌 Every time you change an agent's config (role, model, budget), a agent_config_revisions record is saved — full history.

🎫 What is an Issue?

An issue is a task or ticket — exactly like GitHub Issues or Jira, but assigned to AI agents. Issues can:

  • Be assigned to one or more agents
  • Have parent/child relationships (subtasks)
  • Include comments, attachments, documents
  • Have statuses: backlog → todo → in_progress → in_review → done
  • Block or be blocked by other issues
  • Require approval before the agent can proceed

🌳 Issue Tree

Issues nest into trees. An agent can create sub-issues and delegate work down the hierarchy.

📋 Build note-taking app
  ├── 📋 Design database schema
  ├── 📋 Build REST API
  │    ├── 📋 Auth endpoints
  │    └── 📋 Notes CRUD
  └── 📋 Build frontend

An entire subtree can be paused with an "issue tree hold" — for example, while waiting for your approval on the strategy.

💓 What is a Heartbeat?

Agents don't run continuously. Instead, they wake up periodically (on a schedule, or when triggered), check if there's work to do, do it, and go back to sleep. This wake/work/sleep cycle is called a heartbeat.

Think of it like an employee checking their inbox every morning.

🌐 What is an Environment?

An environment is where the agent actually runs. Options include:

  • Local — On your own machine
  • Remote — On a cloud server
  • Plugin-managed — Via a custom provider

When an agent needs to run, Paperclip acquires an environment lease — like reserving a desk for the agent to sit at while it works.

📁 Execution Workspaces

Each run gets a workspace — a directory with the code. Two strategies:

📂 project_primary
Agent works in the main project directory. Simple, but agents can conflict.
🌿 git_worktree
Each issue gets its own git branch + worktree. Multiple agents work in parallel safely.

💰 Budget System

Every agent has a monthly token/cost budget. Paperclip tracks every API call and stops agents when they hit the limit.

🤖 CEO Agent
28%
🤖 CTO Agent
72%
🤖 Eng Agent 1
100%
🤖 Eng Agent 2
45%

⚠️ When Eng Agent 1 hits 100%, Paperclip creates a budget_incident, pauses the agent, and logs the event. No runaway costs.

⏰ What are Routines?

Routines are cron-style scheduled tasks that automatically trigger agent wakeups. Think of them as recurring alarms for your AI employees.

🌅
Daily Standup
Every weekday at 9am — CEO agent reviews all open issues and reprioritises
0 9 * * 1-5
📊
Weekly Report
Every Friday at 5pm — CMO agent sends marketing performance summary
0 17 * * 5
🔍
Bug Monitor
Every hour — Engineer checks error logs and creates issues for new bugs
0 * * * *

🎯 Goals — The North Star

Every company in Paperclip has a goal — a plain-English mission statement. Every issue, every task, every agent run traces back to this goal.

Company Goal
"Build the #1 AI note-taking app and reach $1M MRR"

When agents work on issues, they receive the goal in their context — so they always know why they're doing what they're doing, not just what to do. This prevents agents from going off-track.

🛡️ Human in the Loop

Approvals are Paperclip's governance system. Certain actions require a human (you) to approve before an agent can proceed.

  • Hiring a new agent
  • Making a major strategy decision
  • Spending over a threshold
  • Any custom approval gate you define

Deep Dive

The Heartbeat — Step by Step

The heartbeat is the core execution loop of Paperclip — the ~1000-line service that actually runs your agents. Click through each step to see what happens under the hood.

1

Trigger — Something wakes the agent up

A routine fires, a comment is posted, or you manually start a run.

Three types of wake triggers exist in agent_wakeup_requests:

  • routine_trigger — A cron job fired (e.g. "every morning at 9am")
  • issue_assignment — An issue was assigned to this agent
  • comment_reply — Someone replied to the agent on an issue thread
// services/heartbeat.ts
const wakeupRequest = await db.select()
  .from(agentWakeupRequests)
  .where(eq(agentWakeupRequests.agentId, agentId))
  .orderBy(asc(agentWakeupRequests.createdAt))
  .limit(1)
2

Acquire a Start Lock

Prevents two simultaneous runs for the same agent.

Before anything else, Paperclip grabs a start lock for the agent. This ensures that if two wakeup requests arrive at the same time, only one run begins.

// services/agent-start-lock.ts
await withAgentStartLock(agentId, async () => {
  // Only ONE heartbeat can be inside here at a time
  await runHeartbeat(...)
})
3

Check Budgets & Permissions

Is this agent paused? Is it over budget? Does it have permission to run?

Paperclip checks multiple gates before spending any money:

  • Is the company active (not paused)?
  • Is the agent's monthly budget exhausted?
  • Is there a pending approval blocking this?
  • Is the issue tree paused (human hold)?
const budgetCheck = await budgets.checkEnforcement({
  companyId,
  scopeType: "agent",
  scopeId: agentId
})
if (budgetCheck.status === "hard_stop") throw new BudgetError()
4

Resolve & Acquire Environment

Which machine will the agent run on? Grab a "lease" for it.

The environment-run-orchestrator.ts service handles this in 3 sub-steps:

  1. Resolve the correct environment (issue config → project policy → agent default → company default)
  2. Probe the environment (is it reachable?)
  3. Acquire a lease — like reserving a timeslot on that machine
const { environment, lease } =
  await orchestrator.acquireEnvironment({
    companyId,
    selectedEnvironmentId: issue.environmentId,
    defaultEnvironmentId: agent.defaultEnvironmentId,
  })
5

Realize the Workspace

Set up the folder / git worktree where the agent will work.

If the project uses git_worktree strategy:

  1. Clone the repo if not already present
  2. Create a new branch like issue/fix-login-bug-abc123
  3. Create a git worktree at a fresh path
  4. Inject env vars: PAPERCLIP_WORKSPACE_DIR, PAPERCLIP_WORKSPACE_BRANCH
const workspace = await realizeExecutionWorkspace({
  repoUrl: project.repoUrl,
  branchTemplate: "issue/{identifier}-{slug}",
  worktreeParentDir: "~/.paperclip/worktrees"
})
6

Spawn the Agent Process

Run the actual AI tool (e.g. claude --print ...) as a child process.

The adapter (e.g. claude-local) spawns the process with a carefully constructed prompt:

  • The issue title + description
  • The agent's role/instructions
  • Previous context (continuation summary)
  • Company goal
  • Wake comments (messages sent to the agent)
const process = spawn("claude", [
  "--print",
  "--model", "claude-sonnet-4-6",
  "--dangerously-skip-permissions",
  prompt
], { cwd: workspace.cwd, env: { ...baseEnv, ...agentEnv } })
7

Stream Output & Record Costs

Capture everything the agent outputs and track token usage in real time.

As the agent runs, Paperclip:

  • Streams stdout chunks to heartbeat_run_events in the DB
  • Publishes live events via WebSocket so the UI updates instantly
  • Writes a cost_event for each LLM call with token counts
  • Enforces the byte cap on stored logs (64KB per chunk)
process.stdout.on("data", async (chunk) => {
  await publishLiveEvent({ companyId, type: "run.output", payload: { chunk } })
  await db.insert(heartbeatRunEvents).values({ runId, chunk })
  await costService.createEvent(companyId, { agentId, inputTokens, outputTokens })
})
8

Wrap Up — Release & Update

Agent finishes. Release the environment lease, update the issue status.

After the process exits:

  1. Classify run result: success / failure / timeout / crashed
  2. Post a summary comment on the issue with what was accomplished
  3. Update issue status (e.g. in_progress → in_review)
  4. Release the environment lease
  5. Check if the issue should continue (schedule next wake)
  6. Log activity for audit trail
await orchestrator.releaseEnvironment({ leaseId: lease.id })
await issueService.update(issueId, { status: "in_review" })
await logActivity({ companyId, action: "heartbeat.completed", metadata: result })

Extensibility

The Plugin System

Plugins let you add new capabilities to Paperclip — custom tools agents can use, scheduled jobs, custom UI, and more. Each plugin runs as its own separate process.

📎 Paperclip Host
⟵ JSON-RPC over stdio ⟶
🔌 Plugin Worker Process

🔄 Plugin Lifecycle

loaded
Plugin process started, manifest validated, tools registered
running
Accepting RPC calls, executing scheduled jobs
crashed
Unexpected exit. Retried with exponential backoff: 1s → 2s → 4s → … → 5min
disabled
After 10 consecutive crashes, plugin gives up and stops restarting

🛡️ Crash Isolation

Since every plugin is a separate child process, a crashing plugin can never take down the Paperclip server. The host detects the crash and restarts with backoff.

Attempt 1: wait 1s → restart
Attempt 2: wait 2s → restart
Attempt 3: wait 4s → restart
...
Attempt 10: wait 5min → give up
Status: disabled

Shutdown sequence: 10s drain → SIGTERM → 5s grace → SIGKILL

Governance

You're the Board — Staying in Control

Paperclip is designed so you can always override, pause, or terminate any agent or process at any time. Here's the governance toolkit:

🛑 Pause Controls

You can pause at different levels:

Pause entire company
All agents stop immediately. No runs start.
Pause single agent
That agent's runs are queued but not started.
Pause issue subtree
Block progress on a branch of issues while you review strategy.

📋 Audit Trail

Everything is logged to activity_log — immutable, append-only records:

heartbeat.started — agt_alice, run_abc
environment.lease_acquired — env_local
issue.status_changed — todo → in_progress
cost.event_created — $0.043, 2400 tokens
heartbeat.completed — success
budget.threshold_exceeded — 72% warning

🔑 Secrets Management

API keys and secrets are stored in company_secrets, versioned, and injected into agent environments at runtime — never stored in plaintext in the agent config.

🌐 Deployment Modes

Choose how locked-down your Paperclip instance is:

local_trusted — single user, no auth needed (personal use)
authenticated — multi-user, login required (team use)
private exposure — blocks requests from unexpected hostnames (SSRF protection)
For the Curious

Inside the Backend Code

Now that you understand the concepts, here's how the TypeScript code is actually structured.

1. The Service Factory Pattern

Every service in Paperclip follows the same pattern — a function that takes db and returns an object of methods. No global singletons (except the event emitter).

pattern used in ALL ~50 services
export function agentService(db: Db) {
  return {
    getById: async (id: string) => { /* ... */ },
    create:  async (data: CreateAgent) => { /* ... */ },
    update:  async (id: string, patch: Partial<Agent>) => { /* ... */ },
    delete:  async (id: string) => { /* ... */ },
  }
}

// Usage anywhere in the codebase:
const agents = agentService(db)
const alice = await agents.getById("agt_alice123")

💡 This pattern makes testing easy — just pass a test database. No mocking global state.

2. Route → Service Architecture

HTTP routes in server/src/routes/ are thin — they validate input and delegate to services.

// routes/agents.ts  (thin — just HTTP wiring)
router.get("/:agentId", async (req, res) => {
  const agent = await agentService(db).getById(req.params.agentId)
  res.json(agent)
})

// services/agents.ts  (fat — all the business logic)
export function agentService(db: Db) {
  return {
    getById: async (id: string) => {
      const agent = await db.select().from(agents)
        .where(eq(agents.id, id))
        .then(rows => rows[0] ?? null)
      if (!agent) throw notFound("Agent not found")
      return agent
    }
  }
}

3. The Express App Structure

The server/src/app.ts file wires everything together. Key middleware order matters:

1
express.json() — parse request body (10MB limit)
2
privateHostnameGuard — block unexpected hostnames
3
actorMiddleware — resolve who's making the request
4
/api/auth — BetterAuth session routes
5
boardMutationGuard — block writes when board is read-only
6
All API routes (/api/companies, /api/agents, etc.)
7
Static React UI (SPA fallback for all /* routes)

📦 On Startup

When Paperclip starts, it runs several initialization tasks:

  • Load all installed plugins (pluginLoader.loadAll())
  • Start plugin job scheduler (cron runner)
  • Start plugin job coordinator (event wiring)
  • Initialize plugin tool dispatcher
  • Begin feedback export flush interval (5s)
  • Start plugin dev watcher (in dev mode)
